あなたのAIエージェントは、50ターン目まで完璧に動いていた。51ターン目から、さっき読んだファイルを忘れ、同じ質問を繰り返し、数分前の文脈を無視した判断を始めた。

これは架空の話ではない。長時間稼働するAIエージェントが必ず直面する「コンテキストウィンドウの壁」だ。2026年3月現在、この問題に対する3つの異なるアプローチが同時に登場し、驚くほど似た結論に収束している。

## 問題の本質: 「記憶がある」≠「記憶が正しい」

AIエージェントのメモリ問題は、単純な容量不足ではない。

100万トークンのコンテキストウィンドウを持つモデルでも、情報を詰め込めば注意力が散漫になる。コストとレイテンシが跳ね上がる。そして最も厄介なのは、古い情報と新しい情報が矛盾したまま共存してしまうことだ。

「メモリがある」ことと「メモリが正しい」ことは、まったく別の問題なのだ。

## 3つのアプローチ、1つの結論

### アプローチ1: 分業で記憶を管理する（AMA）

香港科技大学のチームが発表したAMA（Adaptive Memory via Multi-Agent Collaboration, arXiv:2601.20352）は、記憶管理を4つの専門エージェントに分業させる。

- Constructor: 会話を構造化して記録する
- Retriever: クエリの意図に応じて適切な粒度の記憶を取り出す
- Judge: 取り出した記憶の整合性を検証する
- Refresher: 矛盾を検知したら記憶を更新する

結果は劇的だった。フルコンテキスト方式と比較してトークン消費を約80%削減しながら、精度は向上。特に「知識更新タスク」では約90%の精度を達成した。

ここで注目すべきはJudgeの存在だ。記憶を取り出すだけでなく、「この記憶は今の文脈と矛盾していないか」を検証するステップが入っている。記憶の「存在」だけでなく「内容の正しさ」まで確認する設計だ。

### アプローチ2: ツールをエージェントの認知に合わせる（Anthropic）

Anthropicが2025年9月に公開した「Writing Effective Tools for AI Agents」は、ツール設計のベストプラクティスを体系化したものだ。

核心的な発見がいくつかある。

まず、ツールは少なく意図的に作るべきだということ。既存のAPIをそのままラップするのではなく、エージェントの認知特性に合わせて再設計する。例えば`list_contacts`（全件返却）より`search_contacts`（検索）の方が、エージェントのコンテキストを無駄に消費しない。

次に、レスポンスフォーマットの制御。`concise`と`detailed`を切り替えられるようにするだけで、トークン消費を約3分の1に削減できた実例がある。

そしてエラーメッセージのプロンプトエンジニアリング。「何が間違っていたか」だけでなく「次にどうすべきか」を返すことで、エージェントの自己修復率が上がる。

### アプローチ3: 圧縮しても取り戻せる設計（Memory Compaction）

Branden Collingsworthが2026年1月に公開した実践ガイドは、長時間稼働エージェントのメモリ圧縮戦略を6段階で整理している。

1. 失敗したツール呼び出しの削除
2. 履歴長の制限
3. 大きな出力の要約
4. ローリングサマリーの維持
5. 古いメッセージの切り詰め
6. 重複排除とパターン折りたたみ

ここで最も重要な概念が「Recall Hints」だ。情報を圧縮・削除する際に、パンくずを残す。「何が消えたか」「どうすれば取り戻せるか」をエージェントに伝える仕組みだ。

さらに、宣言的知識（事実・決定の根拠）はRAGに、手続き的知識（成功した手順）はスキルドキュメントに統合することで、セッションを跨いだ学習が可能になる。

## 3つに共通する構造

一見バラバラに見える3つのアプローチだが、共通する構造がある。

- AMA: メモリが**存在する** → メモリが**整合的か**（Judge）
- Anthropic: ツールが**存在する** → ツールがエージェントに**適切か**（評価駆動）
- Compaction: 情報が**圧縮された** → 圧縮後も**取り戻せるか**（Recall Hints）

いずれも「存在するかどうか」で止まらず、「内容が正しいか・機能しているか」まで検証するステップを設計に組み込んでいる。

これは偶然の一致ではない。Chris Argyrisが1977年に提唱したシングルループ学習（エラーを修正する）とダブルループ学習（前提を問い直す）の区別と同じ構造だ。存在チェックはシングルループ、内容チェックはダブルループに対応する。

## 失敗談: 「要約したら大事なことが消えた」

Memory Compactionガイドが正直に認めている失敗パターンが示唆的だ。

データベースマイグレーション計画を要約した際、「外部キーはテーブル作成後に作る」という制約が消えた。エージェントはそのまま進行し、マイグレーションが失敗。デバッグに数時間かかった。監査ログを見ると、元の計画は正しかった。

要約という「圧縮」が、重要な制約という「内容」を破壊したのだ。

これはAMAのRefresherが解決しようとしている問題でもある。記憶を更新する際に、削除ではなく「状態の修正」をデフォルトにする。完全に消すのは、ユーザーが明示的に忘れるよう指示した場合か、保持期限を超えた場合のみ。

## 実践: 今日から試せること

これらの知見を自分のエージェントシステムに適用するなら、以下の3つから始められる。

**1. 記憶の整合性チェックを入れる**

記憶を取り出した後、使う前に「この情報は最新か」「現在の文脈と矛盾しないか」を1ステップ挟む。AMAのJudgeの簡易版だ。

**2. ツールの出力にフォーマット制御を入れる**

`response_format`パラメータを追加し、`concise`（要約）と`detailed`（詳細）を切り替えられるようにする。Anthropicの実例では、これだけでトークン消費が約3分の1になった。

**3. 圧縮時にRecall Hintsを残す**

情報を削除・圧縮する際に「何が消えたか」「どこを見れば取り戻せるか」を1行残す。完全な削除より、取り戻し経路付きの圧縮の方が、長期的な信頼性が高い。

## 限界と反論

これらのアプローチにも限界はある。

AMAのマルチエージェント設計は、静的な検索ベースラインと比較して計算オーバーヘッドが増える。小規模なモデルでは効率が落ちる可能性がある。

Anthropicのツール設計原則は、Claude系モデルでの検証が中心であり、他のモデルでの再現性は未検証の部分がある。ツール名のプレフィックス/サフィックスの効果がモデルによって異なるという報告もある。

Memory Compactionの各戦略は、圧縮の順序やタイミングによって非決定的な結果を生む可能性がある。同じセッションでも、圧縮のタイミングが違えばエージェントの振る舞いが変わりうる。

## 展望: 記憶が「進化」するエージェントへ

Microsoft Researchが2026年3月に発表したAgent Lightningは、エージェントが経験から強化学習で自動的に改善するフレームワークだ。1.5Bパラメータの小さなモデルが、わずか8エポックの学習でGPT-4シリーズと同等の性能を達成した事例がある。

記憶の管理（AMA）、ツールの最適化（Anthropic）、圧縮と保存（Compaction）、そして経験からの学習（Agent Lightning）。これらが統合されたとき、エージェントは「毎回ゼロから始める」存在から「経験を積んで成長する」存在に変わる。

問いを残して終わりたい。

あなたのエージェントは、「覚えている」だけだろうか。それとも「正しく覚えている」だろうか。その違いを検証する仕組みは、設計に組み込まれているだろうか。
